# 2025年工作总结

今年是被AI深度影响的一年，从一开始的了解到逐渐深度使用过程也是颇有反复。刚开始去接触和使用AI，发现可能还没自己手码的快。因为使用的场景都是自身能力能覆盖的日常工作场景，所以在写一大堆提示词的时候感觉很麻烦，还不如自己手写算了。这过程中偶尔还会遇到知识盲区外的功能开发，这时候去用就会发现真TMD的好用。所以这一阶段更多的是把AI当做一个工具、开发助手，相当于给自己配备了一个资深且全能的专家。

后面随着技术社区里越来越多的使用体验与感受的反馈，看到了更多通过使用AI从零到一的项目经历，这时候认知已经在悄然发生改变。意识到还是需要深度使用下去，全面拥抱AI。后面再面对AI时，看法慢慢改变，即使是日常再普通不过的需求迭代，都会去写繁复的提示词。在用的过程中也确实理解了团队内某些同事的代码为什么会出现各种把简单事情复杂化。因为在深度使用AI的过程中发现，AI通常不会考虑后续的维护成本。那么它的关注点更多是在怎么解决问题上，而不是抽象问题的本质，让后续维护更敏捷。

针对这种简单逻辑复杂化的现象，正在尝试通过定义项目规则去规避，还需一段时间的使用才能明确这种方式是否可以避免简单逻辑复杂化的现象。

> 项目规则是cursor每次coding时都会读一遍的文档

### 日常工作

今年在工作中，把自己定位于工具人，公司业务的支撑人员，认真履行自己的职责，努力把事情做好。不过也确实存在做某些需求时，越做情绪越失控的情况。比如印象最深的是Wooacry 首页改版的需求。首页改版可以理解需要随着业务的发展需要对外、对客户透出该项目目前的功能重点以及发展理念的，我本身是不抗拒该需求的。但是验收阶段频繁的去改字重、字体大小是实在不理解，所以这种频繁的改动在我目前的认知理解中就是无用功，一个字体的字重或大小到底会影响什么呢？我们的主要营收是受字重、字体大小、间距等这些样式影响的吗。当时很不理解，一个部门领导为什么去扣这些细节，她的关注点不应该业务增长、用户增长、营收增长上面吗。这种样式不应该是给更专业的设计部门来处理吗。所以，这种频繁扣细节的事情，我是很不理解，也久久无法释怀。本质是部门通力合作的事情，为什么会发展成个人喜好去影响这件事的发展。

综上所述，还是心态不够稳重，应该被这些事情而去影响自己的心态。毕竟工作的性质是工具人，领导怎么说只能这么做，太较真了就是和自己过不去。

另外印象较深的是和同组的某同事一起合作写新项目的事情，如果经常延期可以说是能力问题，但是你延期了却不主动寻求解决方案，也不复盘为什么会延期，这就是态度问题了。至少我是没有从他身上看出来他对待工作有多负责，往往只是做浅层的一些事情，而不关心这次的改动会不会对其他功能有影响。所以往往到了验收阶段或上线一段时间内就会发现问题。如果说对比的话，比他早一点过来的同事就是反例。虽然前期因为不太熟悉业务情况、代码背景，但人家起码是真在认真学习以及主动加班去尽量不让功能迭代延期上线。

一开始能力差一点，团队内大家都是愿意一起携手共进的。但是你能力相对而言弱一些，工作态度也体现不出来，那让大家该怎么去帮助他呢。和他提一些反馈反而可能会让他认为在针对他，那么还有单独和他提的必要嘛。所以，对待某些同事而言，那就是公事公办。

在这里面，我是有些看不惯这种混日子的同事的。但是世界是多样性的，所以也必然会有形形色色的人，所以还是要有容人之量。公司层面说他行，那他确实就是行。我定义不了他，这些也只是我个人对他的看法，短期内也不太可能扭转的过来。除非他确实有让我感受到实实在在的改变，那印象也会有所改观。

### 新一年的展望

比教期望自己能扩展边界能力，随着AI的发展，已经在模糊软件开发中的各种岗位了。以前会分的很细是因为个人的精力有限，需要深耕某一方向，然后大家通力合作才能把事情做好。目前AI的发展，给我的感受是应该解放视野，让每个人都成为超级个体，AI做每个人的全能顾问。
